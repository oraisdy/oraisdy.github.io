Android中的MVP与MVVM

发表于 2017-05-27

今天发现了一篇长文：GUI 应用程序架构的十年变迁：MVC、MVP、MVVM、Unidirectional、Clean，对于MV*模式解释的十分详尽。我对文中一些内容进行了笔记，并且将Android部分结合代码希望能更深入的理解。

所有代码来自Github仓库: https://github.com/ivacf/archi

MV*模式中的三种元素

Model：负责操纵数据。

View：负责将渲染数据展示给用户和响应用户界面操作。Activity和Fragment都属于View层。

Controller / Presenter / ViewModel：作为View与Model交互的中间纽带，处理与用户交互的负责逻辑。

经典Android程序

在经典的Android程序中，Activity负责调用model中的方法来处理数据和更新界面，它的职责过于庞大和复杂，并且难以测试。

MVC存在的问题

MVC模式在一些小型项目或者简单的界面上仍旧有极大的可用性，但是在现代富客户端开发中导致职责分割不明确、功能模块重用性、View的组合性较差。

MVP

从 MVC 到 MVP 的变迁完成了对于 View 与 Model 的解耦，改进了职责分配与可测试性。

使用情景

  • 用户产生输入
  • View监听到Action并进行转化发送给Presenter处理
  • Presenter调用Model更新数据
  • Model更新数据后通知和返回数据给Presenter
  • Presenter把更新后的数据交给View显示

MVP的缺陷

  • View需要了解Presenter的细节，将用户交互进行转化并发送给Presenter处理
  • Model需要了解Presenter的细节，在完成更新后把新数据传递给Presenter

MVVM

而从 MVP 到 MVVM，添加了View与ViewModel之间的数据绑定，使得 View 完全的无状态化。

使用情景

  • 用户交互输入

  • View将数据直接传送给ViewModel，ViewModel保存这些状态数据

  • 在有需要的情况下，ViewModel会将数据传送给Model

  • Model在更新完成之后通知ViewModel

  • ViewModel从Model中获取最新的模型，并且更新自己的数据状态

  • View根据最新的ViewModel的数据进行重新渲染

MVVM的缺陷

将View层轻量化、无状态化，不需要保存任何状态信息。同时ViewModel与View解耦，便于对View进行测试。MVVM存在的问题是：

  • ViewModel的职责复杂
  • 由于View和ViewModel的双向绑定，数据可以被任意的修改，数据在多个ViewModel可能存在冗余与不一致问题
